# 医疗AI辅助诊断应用指南（完整版）

## 1. AI辅助诊断概述

### 1.1 应用范围

#### 影像学辅助诊断
- 胸部X线/CT：肺炎、肺结节、肺癌筛查
- 头颅CT/MRI：脑出血、脑梗死、颅内肿瘤
- 骨骼X线：骨折检测、骨龄评估
- 心电图：心律失常、心肌缺血自动分析
- 眼底照片：糖尿病视网膜病变筛查

#### 临床决策支持
- 智能分诊与导诊
- 用药安全审核（药物相互作用、禁忌证）
- 临床路径推荐
- 检查检验结果解读
- 病历质量控制

#### 自然语言处理应用
- 电子病历结构化
- 医学文献检索与摘要
- 患者主诉智能分析
- 医患对话记录整理

### 1.2 AI诊断的局限性
- 辅助参考：AI诊断仅为辅助工具，不能替代医生最终判断
- 罕见病：对罕见病、不典型病例诊断准确率有限
- 数据依赖：训练数据质量和完整性直接影响模型性能
- 体格检查：无法替代医生的体格检查和问诊
- 情境理解：难以理解复杂的社会心理因素
- 持续学习：需要定期更新模型以适应新知识

### 1.3 AI辅助诊断伦理要求
- 知情同意：患者有权知道AI参与了诊疗过程
- 隐私保护：患者数据需脱敏处理
- 透明性：AI决策过程应可解释
- 责任界定：AI辅助诊断的医疗责任归属
- 公平性：避免算法偏见导致的诊疗差异

## 2. 智能分诊系统

### 2.1 分诊算法架构

#### 数据输入层
- 患者主诉（自然语言文本）
- 症状持续时间
- 伴随症状列表
- 基本生命体征（如有）
- 年龄、性别等基本信息

#### 特征提取层
- 症状分词与标准化（医学术语映射）
- 症状权重计算
- 时间因素评估
- 紧急程度判断

#### 决策推理层
- 症状-疾病概率映射
- 科室推荐排序
- 紧急程度分级
- 置信度评估

#### 输出层
- 推荐科室列表（按置信度排序）
- 紧急程度分级（I-V级）
- 推荐理由
- 建议的下一步行动

### 2.2 科室推荐逻辑

#### 症状-科室映射表

| 症状关键词 | 主推荐科室 | 次推荐科室 | 置信度权重 |
|-----------|-----------|-----------|-----------|
| 胸痛、心悸、气短、胸闷 | 心血管内科 | 呼吸内科 | 0.85 |
| 咳嗽、咳痰、呼吸困难、咯血 | 呼吸内科 | 心血管内科 | 0.80 |
| 腹痛、腹泻、恶心、呕吐、黑便 | 消化内科 | 外科 | 0.82 |
| 头痛、头晕、肢体麻木、无力 | 神经内科 | 心血管内科 | 0.78 |
| 多饮、多尿、体重下降、乏力 | 内分泌科 | - | 0.88 |
| 骨折、关节痛、活动受限、肿胀 | 骨科 | - | 0.90 |
| 皮疹、瘙痒、皮肤红肿、脱屑 | 皮肤科 | - | 0.85 |
| 牙痛、牙龈出血、口腔溃疡 | 口腔科 | - | 0.92 |
| 视力模糊、眼痛、眼红 | 眼科 | 神经内科 | 0.88 |
| 耳鸣、听力下降、鼻塞、鼻出血 | 耳鼻喉科 | - | 0.85 |
| 尿频、尿急、尿痛、血尿 | 泌尿外科 | 肾内科 | 0.87 |
| 关节痛、皮下结节、晨僵 | 风湿免疫科 | 骨科 | 0.80 |
| 失眠、焦虑、情绪低落 | 精神心理科 | 神经内科 | 0.75 |
| 发热、乏力、体重下降 | 感染科 | 血液科 | 0.70 |
| 消瘦、多汗、手抖 | 内分泌科 | - | 0.82 |

#### 综合评分算法
```
总分 = Σ(症状权重 × 科室匹配度) × 紧急程度系数

紧急程度系数：
- I级（濒危）：×2.0
- II级（危重）：×1.5
- III级（急症）：×1.2
- IV级（亚急症）：×1.0
- V级（非急症）：×0.8
```

### 2.3 紧急程度评估

#### 红色（I级-濒危）
**识别特征：**
- 胸痛伴大汗、面色苍白、濒死感
- 意识丧失或GCS≤8分
- 严重呼吸困难（端坐呼吸、大汗淋漓）
- 大量活动性出血
- 严重外伤（多发伤、复合伤）
- 抽搐持续状态

**处理：** 立即进入抢救室，心电监护，建立静脉通路

#### 橙色（II级-危重）
**识别特征：**
- 持续剧烈胸痛（>20分钟，硝酸甘油不缓解）
- 急性肢体无力/偏瘫
- 高热伴意识改变（>39.5°C+精神差）
- 严重过敏反应（荨麻疹+呼吸困难）
- 急性腹痛伴休克征象

**处理：** 10分钟内接诊，优先安排检查

#### 黄色（III级-急症）
**识别特征：**
- 中度疼痛（VAS 4-6分）
- 持续发热（38.5-39.5°C）
- 中度呼吸困难（可平卧但活动受限）
- 中度脱水（皮肤弹性差、口渴、尿少）

**处理：** 30分钟内接诊

#### 绿色（IV-V级）
**识别特征：**
- 轻度症状（轻微疼痛、低热）
- 慢性病复诊
- 体检咨询
- 非紧急皮肤问题

**处理：** 60分钟内接诊或转门诊

### 2.4 分诊质量控制
- 准确率评估：与专家分诊结果对比
- 敏感度评估：危重患者识别率
- 响应时间评估：从输入到输出的时间
- 用户满意度评估：患者和医生反馈

## 3. AI分诊算法技术实现

### 3.1 算法选型对比

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 规则引擎 | 可解释性强、确定性输出 | 覆盖率有限、维护成本高 | 基础分诊规则 |
| 贝叶斯网络 | 概率推理、处理不确定性 | 需要先验概率、条件独立假设 | 疾病概率推断 |
| 随机森林 | 特征重要性、抗过拟合 | 黑盒、需大量标注数据 | 症状-科室分类 |
| 深度学习（BERT） | 语义理解能力强 | 计算资源需求大、可解释性差 | 主诉文本分析 |
| 大模型（LLM） | 综合推理、少样本学习 | 幻觉风险、延迟较高 | 综合分诊决策 |

### 3.2 基于规则的分诊引擎

#### 规则定义格式
```json
{
  "rule_id": "TRIAGE_CHEST_PAIN_001",
  "name": "急性胸痛分诊规则",
  "priority": 1,
  "conditions": {
    "symptom": ["胸痛", "胸闷", "心悸"],
    "time_range": "<=30min",
    "severity": "严重",
    "accompanying": ["大汗", "恶心", "呼吸困难"],
    "vital_signs": {
      "systolic_bp": {"op": "<", "value": 90},
      "heart_rate": {"op": ">", "value": 120}
    }
  },
  "actions": {
    "triage_level": "I",
    "department": "心血管内科",
    "time_to_doctor": 0,
    "priority_checks": ["心电图", "心肌酶", "胸片"]
  },
  "confidence": 0.92
}
```

#### 规则冲突解决策略
1. 优先级规则：高优先级规则覆盖低优先级
2. 置信度规则：选择置信度最高的匹配规则
3. 特异性规则：更具体的规则优先于一般规则
4. 时间规则：更紧急的情况优先处理

### 3.3 基于NLP的症状提取

#### 分词与实体识别流程
```
原始文本："我胸口疼了两个小时，还伴有呼吸困难和出汗"
    ↓
分词：[我] [胸口] [疼] [了] [两个] [小时] [，] [还] [伴有]
      [呼吸困难] [和] [出汗]
    ↓
实体识别：
  - 症状1: 胸口疼 → "胸痛"（标准化）
  - 症状2: 呼吸困难 → "呼吸困难"
  - 症状3: 出汗 → "多汗"
  ↓
时间提取：两个小时 → "2小时"
    ↓
严重程度推断：多个严重症状共存 → "重度"
```

#### 医学术语标准化映射

| 患者表述 | 标准术语 | 同义词列表 |
|---------|---------|-----------|
| 胸口疼、心口疼、胸痛 | 胸痛 | 胸部不适、胸闷、心前区疼痛 |
| 拉肚子、闹肚子、泻肚子 | 腹泻 | 水样便、稀便、拉水 |
| 头疼、脑袋疼、偏头疼 | 头痛 | 头胀、头昏、搏动性头痛 |
| 喘不上气、憋气、气短 | 呼吸困难 | 气促、憋闷、喘息 |
| 心慌、心跳快、心脏跳得厉害 | 心悸 | 心跳加速、心律不齐 |
| 胃不舒服、恶心、想吐 | 恶心/呕吐 | 反胃、干呕、胃部不适 |
| 手脚麻、发麻、发木 | 麻木 | 感觉减退、刺痛 |

### 3.4 多模态特征融合

#### 特征类型与处理

| 特征类型 | 数据来源 | 处理方法 | 权重 |
|---------|---------|---------|------|
| 文本特征 | 主诉、病史 | BERT编码 | 0.35 |
| 生命体征 | 体征数据 | 标准化+归一化 | 0.30 |
| 人口学特征 | 年龄、性别 | One-Hot编码 | 0.10 |
| 时间特征 | 症状持续时间 | 对数变换 | 0.10 |
| 既往史 | 慢性病、用药 | 编码+加权 | 0.15 |

#### 融合策略
```
特征融合方法：
1. 早期融合：所有特征拼接后输入模型
2. 中间融合：各模态分别编码后在中间层融合
3. 晚期融合：各模态独立预测后加权平均

推荐方案：中间融合（兼顾特征交互和独立性）
```

### 3.5 置信度校准

#### 校准方法
- Platt Scaling：使用逻辑回归校准输出概率
- Temperature Scaling：调整softmax温度参数
- Isotonic Regression：非参数校准方法

#### 置信度阈值设置

| 置信度范围 | 处理策略 |
|-----------|---------|
| ≥0.90 | 直接推荐科室 |
| 0.70-0.89 | 推荐科室+建议复核 |
| 0.50-0.69 | 推荐多个科室供选择 |
| <0.50 | 标记为不确定，转人工分诊 |

## 4. RAG检索优化策略

### 4.1 RAG架构概述

```
┌─────────────────────────────────────────────┐
│                   用户查询                     │
│     "患者胸痛2小时，伴大汗，心电图ST段抬高"     │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│              查询理解与优化层                   │
│  - 意图识别：急诊分诊/疾病诊断/用药咨询         │
│  - 关键词提取：胸痛、ST段抬高、急性心梗          │
│  - 查询扩展：同义词、相关术语                    │
└──────────────────┬──────────────────────────┘
                   ↓
┌──────────────────┬──────────────────────────┐
│    语义检索（向量数据库）   │   关键词检索（BM25）    │
│    Top-K=10                │   Top-K=10            │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│              混合检索与重排序                   │
│  - 分数融合（加权平均）                        │
│  - 交叉编码器重排序                            │
│  - 去重与相关性过滤                            │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│              上下文构建                        │
│  - Top-N文档拼接                              │
│  - 相关性标注                                 │
│  - Token预算控制                              │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│              大模型推理                        │
│  - Prompt构建 + 检索上下文                    │
│  - 结构化JSON输出                             │
│  - 置信度评估                                 │
└─────────────────────────────────────────────┘
```

### 4.2 向量化处理流程

```
原始文档
    ↓
文本预处理（去噪、格式化）
    ↓
文本分块（按语义段落/固定长度）
    ↓
Embedding向量化（使用DeepSeek Embedding模型）
    ↓
存储至向量数据库（Weaviate/Milvus）
    ↓
建立索引（支持语义检索和关键词检索）
```

### 4.3 文档分块策略

| 分块方法 | 描述 | 优点 | 缺点 | 适用场景 |
|---------|------|------|------|---------|
| 固定长度分块 | 按字符/Token数切分 | 简单高效 | 可能切断语义 | 通用文档 |
| 语义分块 | 按段落/章节切分 | 保持语义完整 | 块大小不均匀 | 结构化文档 |
| 滑动窗口 | 重叠窗口切分 | 减少信息丢失 | 冗余度高 | 长文档 |
| 递归分块 | 层级切分 | 兼顾效率和语义 | 实现复杂 | 混合文档 |
| 智能分块 | 基于LLM语义切分 | 最佳语义保持 | 成本最高 | 高价值文档 |

#### 推荐分块参数
```
医疗知识库文档分块建议：
- 块大小：512-1024 tokens
- 重叠：128-256 tokens
- 分块方法：语义分块（按章节/表格）
- 元数据：文档标题、章节标题、更新日期
```

### 4.4 查询优化技术

#### 查询扩展
| 技术 | 描述 | 示例 |
|------|------|------|
| 同义词扩展 | 添加同义词 | "胸痛" → "胸痛 OR 胸部疼痛 OR 心前区痛" |
| 上下位词扩展 | 添加上位/下位词 | "肺炎" → "肺炎 OR 社区获得性肺炎 OR 细菌性肺炎" |
| 关联词扩展 | 添加相关术语 | "心电图ST段抬高" → "STEMI OR 急性心梗" |
| 纠错扩展 | 修正拼写错误 | "糖屎病" → "糖尿病" |

#### 查询重写
```
原始查询："老头肚子疼拉稀"
    ↓
LLM查询重写：
"老年男性患者，主诉腹痛伴腹泻，需要鉴别诊断：
1. 急性胃肠炎
2. 食物中毒
3. 炎症性肠病
4. 肠道感染
请检索相关诊断标准和鉴别要点。"
```

### 4.5 检索策略详解

#### 语义检索（Semantic Search）
- 基于向量余弦相似度
- 能理解语义相近的表述
- 适合模糊查询和开放性问题

#### 关键词检索（Keyword Search）
- 基于BM25算法
- 精确匹配关键词
- 适合查找特定药物、疾病名称

#### 混合检索（Hybrid Search）
- 结合语义和关键词检索
- 综合排序，提高准确率
- 适用于复杂查询场景

#### RRF（Reciprocal Rank Fusion）融合公式
```
RRF_score(d) = Σ 1/(k + rank_i(d))

其中：
- k = 60（常数）
- rank_i(d) = 文档d在第i个检索器中的排名
- 各检索器权重相等
```

### 4.6 重排序策略

#### Cross-Encoder重排序
```
步骤1：Bi-Encoder检索Top-50候选文档
步骤2：Cross-Encoder对(query, doc)对打分
步骤3：按Cross-Encoder分数重新排序
步骤4：取Top-5作为最终上下文
```

#### 重排序模型对比

| 模型 | 准确率 | 速度 | 适用场景 |
|------|--------|------|---------|
| bge-reranker-base | 高 | 中 | 通用重排序 |
| bge-reranker-large | 更高 | 慢 | 高精度需求 |
| cross-encoder/ms-marco | 高 | 中 | 英文文档 |
| 自训练医疗重排序模型 | 最高 | 可调 | 医疗专用 |

### 4.7 检索质量评估

| 指标 | 定义 | 目标值 |
|------|------|--------|
| Recall@5 | Top-5中包含相关文档的比例 | ≥80% |
| Recall@10 | Top-10中包含相关文档的比例 | ≥90% |
| MRR | 第一个相关文档的排名倒数 | ≥0.7 |
| NDCG@5 | 前5个结果的归一化折损累积增益 | ≥0.75 |
| 响应时间 | 检索+重排序总耗时 | ≤2秒 |

### 4.8 检索质量优化清单
- 查询扩展：自动扩展同义词
- 重排序：基于相关性二次排序
- 过滤：按文档类型、发布时间过滤
- 上下文窗口：返回相关段落的前后文
- 反馈循环：利用用户点击数据优化检索
- 缓存机制：热门查询结果缓存
- 分层索引：按疾病/科室建立子索引

## 5. 大模型Prompt工程最佳实践

### 5.1 Prompt设计原则

| 原则 | 描述 | 示例 |
|------|------|------|
| 角色设定 | 明确AI角色 | "你是一位经验丰富的急诊科医生" |
| 任务明确 | 清晰描述任务 | "请根据以下症状进行急诊分诊" |
| 格式规范 | 指定输出格式 | "请以JSON格式输出结果" |
| 约束条件 | 设定边界 | "不要给出最终诊断，仅提供分诊建议" |
| 示例引导 | Few-shot学习 | "参考以下示例进行输出" |

### 5.2 分诊Prompt模板

```
## 角色
你是一位专业的急诊分诊AI助手，具有10年急诊工作经验。
你的任务是根据患者信息进行急诊分诊分级和科室推荐。

## 输入信息
- 患者主诉：{chief_complaint}
- 症状持续时间：{duration}
- 伴随症状：{accompanying_symptoms}
- 生命体征：{vital_signs}
- 年龄：{age}
- 性别：{gender}
- 既往史：{medical_history}
```

### 5.5 Prompt优化技巧

#### 1. Chain-of-Thought（思维链）
```
请逐步分析：
1. 首先，识别患者的主要症状
2. 然后，根据症状判断可能的疾病
3. 接着，评估疾病的紧急程度
4. 最后，给出分诊建议和理由
```

#### 2. Self-Consistency（自一致性）
```
请从3个不同角度分析这个病例，然后综合给出结论：
角度1：从症状角度分析
角度2：从病史角度分析
角度3：从风险因素角度分析
综合结论：基于以上分析...
```

#### 3. ReAct（推理+行动）
```
思考：患者胸痛伴大汗，首先需要排除急性冠脉综合征
行动：检查心电图和心肌酶结果
观察：心电图ST段抬高，肌钙蛋白升高
思考：符合STEMI诊断标准
行动：建议立即启动胸痛绿色通道
```

### 5.6 医疗Prompt安全指南

| 风险 | 防护措施 |
|------|---------|
| AI幻觉（编造事实） | 要求引用知识库来源，设置置信度阈值 |
| 过度自信 | 加入不确定性表达，要求"仅供参考" |
| 遗漏关键信息 | 结构化输入模板，必填字段校验 |
| 格式不一致 | 严格JSON Schema验证 |
| 延迟过高 | 设置超时机制，降级到简化Prompt |

## 6. Dify Workflow设计模式

### 6.1 Dify平台概述

Dify是一个开源的LLM应用开发平台，提供：
- 可视化Workflow编排
- RAG知识库管理
- 多模型支持（DeepSeek、OpenAI等）
- API接口暴露
- 监控和日志

### 6.2 三种应用类型对比

| 类型 | 特点 | 适用场景 | 开发复杂度 |
|------|------|---------|-----------|
| 聊天助手 | 多轮对话、上下文记忆 | 患者问诊 | 低 |
| 文本生成 | 单次输入输出 | 病历生成 | 低 |
| Workflow | 复杂流程编排、多节点 | 智能分诊 | 中-高 |

### 6.3 智能分诊Workflow设计

#### 节点1：开始节点（输入）
```yaml
节点类型: Start
输入变量:
  - chief_complaint: string (患者主诉)
  - duration: string (症状持续时间)
  - symptoms: string[] (伴随症状列表)
  - vital_signs: object (生命体征)
  - age: integer (年龄)
  - gender: string (性别)
```

#### 节点2：查询理解（LLM节点）
```yaml
节点类型: LLM
模型: DeepSeek-Chat

  主诉：{{chief_complaint}}
  持续时间：{{duration}}
  伴随症状：{{symptoms}}
  
  输出JSON格式：
  {
    "key_symptoms": ["关键症状列表"],
    "urgency_signals": ["紧急信号列表"],
    "severity_estimate": "轻/中/重/危重"
  }
```

#### 节点3：知识库检索（知识库节点）
```yaml
节点类型: Knowledge Retrieval
知识库: 医疗知识库
检索模式: 混合检索
Top-K: 5
相关性阈值: 0.6
查询: "{{chief_complaint}} {{key_symptoms}}"
```

#### 节点4：分诊决策（LLM节点）
```yaml
节点类型: LLM
模型: DeepSeek-Chat

  患者信息：
  - 主诉：{{chief_complaint}}
  - 关键症状：{{key_symptoms}}

  - 年龄：{{age}}
  - 性别：{{gender}}
  - 生命体征：{{vital_signs}}
  
  参考知识：
  {{#context#}}
  
  请进行急诊分诊，输出JSON格式：
  {
    "triage_level": "I-V",
    "confidence": 0.0-1.0,
    "department": "推荐科室",
    "reason": "分诊理由",
    "actions": ["建议操作"]
  }
```

#### 节点5：条件分支
```yaml
节点类型: If/Else
条件:
  - 条件1: triage_level == "I" → 立即通知抢救团队
  - 条件2: triage_level == "II" → 通知急诊医生
  - 条件3: triage_level == "III" → 安排候诊
  - 默认: 建议转门诊
```

#### 节点6：输出格式化（代码节点）
```yaml
节点类型: Code
语言: Python3
代码: |
  import json
  
  triage_result = inputs["triage_result"]
  
  output = {
    "success": True,
    "data": {
      "triage_level": triage_result["triage_level"],
      "triage_level_name": get_level_name(triage_result["triage_level"]),
      "department": triage_result["department"],
      "confidence": triage_result["confidence"],
      "reason": triage_result["reason"],
      "actions": triage_result["actions"],
      "timestamp": get_current_time()

  return {"result": json.dumps(output, ensure_ascii=False)}
```

### 8.4 算法公平性

#### 偏见类型

| 偏见类型 | 描述 | 检测方法 |
|---------|------|---------|
| 选择偏见 | 训练数据不代表总体 | 数据分布分析 |
| 测量偏见 | 特征测量不准确 | 交叉验证 |
| 标签偏见 | 标注不一致 | 标注者一致性分析 |
| 聚合偏见 | 忽略群体差异 | 分层性能分析 |
| 部署偏见 | 部署环境与训练环境不同 | A/B测试 |

#### 公平性指标

| 指标 | 定义 | 公平性标准 |
|------|------|-----------|
| 统计均等性 | 不同群体阳性率相同 | 差异<0.1 |
| 机会均等性 | 不同群体真阳性率相同 | 差异<0.05 |
| 预测均等性 | 不同群体假阳性率相同 | 差异<0.05 |
| 校准公平性 | 不同群体预测概率校准一致 | 校准误差<0.05 |

### 8.5 责任界定

```
医疗AI责任链：

患者 → 医生（最终决策责任）
         ↑
         AI系统（辅助工具责任）
         ↑
         AI开发方（系统质量责任）
         ↑
         医疗机构（采购使用责任）

关键原则：
1. AI是辅助工具，不承担医疗责任
2. 医生对AI辅助下的诊疗决策负最终责任
3. AI开发方对系统质量和安全性负责
4. 医疗机构对AI的适当使用负责
```

## 9. 模型评估指标

### 9.1 诊断准确性

| 指标 | 定义 | 计算公式 |
|------|------|---------|
| 敏感度（Sensitivity） | 真阳性率 | TP/(TP+FN) |
| 特异度（Specificity） | 真阴性率 | TN/(TN+FP) |
| 准确率（Accuracy） | 整体正确率 | (TP+TN)/(TP+TN+FP+FN) |
| 精确率（Precision） | 阳性预测值 | TP/(TP+FP) |
| F1分数 | 精确率和敏感度的调和平均 | 2×P×R/(P+R) |
| AUC-ROC | 综合判别能力 | ROC曲线下面积 |

### 9.2 临床实用性

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 响应时间 | <3秒 | 从输入到输出的时间 |
| 可解释性 | 高 | 诊断依据是否清晰 |
| 用户接受度 | ≥80% | 医生满意度调查 |
| 安全性 | 0不良事件 | AI建议导致的不良事件 |
| 覆盖率 | ≥95% | 能处理的病例比例 |

### 9.3 分诊准确性评估

| 评估维度 | 方法 | 目标 |
|---------|------|------|
| I级识别敏感度 | 与专家分诊对比 | ≥98% |
| II级识别敏感度 | 与专家分诊对比 | ≥95% |
| 分诊一致性 | Kappa系数 | ≥0.8 |
| 过度分诊率 | 统计V级→I级比例 | <5% |
| 欠分诊率 | 统计I级→V级比例 | <1% |

### 9.4 RAG检索质量评估

| 指标 | 定义 | 目标值 |
|------|------|--------|
| Recall@5 | Top-5召回率 | ≥80% |
| Recall@10 | Top-10召回率 | ≥90% |
| MRR | 平均倒数排名 | ≥0.7 |
| NDCG@5 | 归一化折损累积增益 | ≥0.75 |
| 延迟 | 检索+重排序耗时 | ≤2s |

### 9.5 持续改进
- 定期收集用户反馈
- 分析错误案例，优化模型
- 更新知识库，补充新指南
- 监控模型性能，及时调整
- A/B测试新版本

## 10. 知识库构建

- 药品相互作用数据

## 11. 智慧云脑平台AI模块设计

### 11.1 系统架构
```
患者端/医生端/管理端
        ↓
    Gateway网关
        ↓
    业务服务层
        ↓
    AI服务层（ai-service）
    ├── 智能分诊模块
    ├── 病历生成模块
    └── 处方审核模块
        ↓
    Dify AI平台
    ├── Workflow引擎
    ├── 知识库（RAG）
    └── 大模型（DeepSeek）
```

### 11.2 API设计

| API端点 | 方法 | 功能 | 输入 | 输出 |
|---------|------|------|------|------|
| /api/ai/triage | POST | 智能分诊 | 患者症状信息 | 分诊结果JSON |
| /api/ai/medical-record | POST | 病历生成 | 对话记录 | 结构化病历 |
| /api/ai/prescription | POST | 处方审核 | 处方+患者信息 | 审核结果 |
| /api/ai/knowledge | GET | 知识检索 | 查询关键词 | 相关文档列表 |

### 11.3 降级策略详解

| 故障场景 | 降级方案 | 降级后表现 |
|---------|---------|-----------|
| Dify完全不可用 | Mock Provider返回预设结果 | 返回基础分诊结果 |
| LLM响应超时（>30s） | 返回简化结果+降级标识 | 分诊准确性降低 |
| 知识库检索失败 | 仅使用LLM自身知识 | 可能遗漏最新指南 |
| 网络部分不可达 | 本地缓存+重试 | 延迟增加 |
| 数据库连接失败 | Redis缓存兜底 | 可能返回旧数据 |

### 11.4 监控指标

| 监控项 | 告警阈值 | 处理措施 |
|--------|---------|---------|
| API响应时间 | P95>5s | 优化Prompt/模型 |
| 错误率 | >1% | 检查服务状态 |
| Dify服务状态 | 不健康 | 自动重启 |
| 知识库大小 | 增长>50% | 审核新增内容 |
| LLM调用量 | QPS>100 | 限流/扩容 |
