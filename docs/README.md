## 待确认问题

1. AI 服务提供方未明确：可选本地 Mock、学校提供接口或第三方大模型 API。
2. 医生账号来源未明确：MVP 默认医生账号由数据库预置，不开放医生注册。
3. 是否需要管理员端未明确：MVP 暂不实现管理员端，仅通过初始化 SQL 维护医生、科室、药品。
4. 是否涉及真实支付、医保、检查检验系统未明确：MVP 不包含支付、医保、LIS/PACS 对接。
5. 药品知识库来源未明确：MVP 使用本地药品基础数据加 AI 审核结果模拟，后续可接知识图谱。
6. 医疗合规边界未明确：MVP 定位为教学实践系统，AI 结果仅作辅助建议，最终由医生确认。

## 基础假设

- 项目为课程实践和演示系统，不用于真实医疗诊疗决策。
- 系统包含患者端和医生端两个主要使用入口。
- 后端基础业务采用 Spring Boot 3；最终交付必须将 AI 能力层抽取为独立微服务，并使用 Spring Cloud Feign 完成服务间调用。
- 数据库默认使用 MySQL 8。
- 前后端通过 RESTful API 通信，接口统一返回 `Result<T>`。
- 登录认证使用 JWT。
- AI 接口尽量返回结构化 JSON；若 AI 服务不可用，系统回退到人工录入流程。

## 建议阅读顺序

1. 先读 `01` 到 `03`，明确项目边界、用户角色和业务闭环。
2. 再读 `04` 到 `08`，完成技术架构、数据库、接口和页面设计对齐。
3. 开发执行以 `09` 为主，测试和部署分别参考 `10`、`11`。
4. 图片中的六项任务全部为必做交付项，开发执行以 `09` 为准，顺序为 WebSocket、流式响应、Pinia 状态机、Prompt、双端分离部署、微服务拆分。
