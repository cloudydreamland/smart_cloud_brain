# 09 开发计划

## 当前目标

优先交付 P0 可演示闭环：患者完成 AI 分诊与预约挂号，医生完成病历与处方，管理员完成基础数据、字典、分诊工作台和 AI 排班发布。

## P0 闭环

1. 患者端
   - 使用 `13800000001 / 123456` 登录。
   - 提交主诉生成 AI 分诊记录。
   - 从金仓 `appointment_slot` 读取可用号源并完成预约。
   - 可取消自己的挂号。
   - 可查看自己的病历和处方。

2. 医生端
   - 使用 `doctor1 / 123456` 或医生手机号登录。
   - 查看当前医生名下挂号队列。
   - 基于挂号生成 AI 病历草稿并确认保存。
   - 基于已保存病历执行处方审核并确认处方。
   - 通过 WebSocket 接收处方风险通知。

3. 管理端
   - 使用 `admin / 123456` 登录。
   - 维护科室、医生、药品、Prompt、知识库和系统字典。
   - 生成 AI 排班建议并发布为医生排班和可预约号源。
   - 查看分诊工作台，支持分配医生和关闭记录。

## 数据策略

- 金仓数据库是演示环境唯一数据源。
- 基础演示数据写入 `sql/kingbase_schema.sql`，包括患者、医生、管理员、科室、药品、Prompt、知识库、字典、排班和号源。
- 挂号、病历、处方、通知等交易数据通过演示流程实时生成。

## 开发顺序

1. 补齐数据库结构和种子数据。
2. 补齐服务层权限校验，保证患者、医生、管理员只能访问允许的数据。
3. 补齐后台排班、字典、分诊工作台 API。
4. 接入患者端、医生端、管理端页面。
5. 执行 Maven 编译和三个 Web 端构建。
6. 按验收清单完成端到端演示。

## 非 P0 范围

- 不接入真实支付、医保、短信、生产审计。
- AI 能力默认使用 mock/规则结果，接口保留真实模型替换空间。
