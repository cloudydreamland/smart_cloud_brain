# 开发计划文档

## 1. 开发目标

项目开发目标以 Word 任务描述中的六个核心任务为主线，完成一条可运行、可演示的诊疗闭环，并把图片中的六项任务全部纳入必做交付。核心闭环如下：

```text
患者注册登录 -> 智能分诊 -> 在线挂号 -> 医生查看患者 -> AI 生成病历 -> AI 处方审核 -> 保存处方 -> 患者查看记录
```

开发原则：

- P0 覆盖项目初始化、传统业务、三大 AI 模块、综合联调和六项必做任务。
- P1 仅保留管理端、知识图谱深度接入等非任务描述强制要求的增强项。
- WebSocket、AI 流式响应、Pinia 状态机、Prompt 工程、双端分离部署、微服务拆分全部是 P0 必做项。
- AI 能力先 Mock，再替换真实接口，避免阻塞主流程。

## 2. 阶段划分

| 阶段 | 任务 | 优先级 | 主要产出 | 验收标准 |
|---|---|---|---|---|
| 第 1 阶段 | 项目初始化 | P0 | 前端脚手架、`diagnosis-service`、`ai-service`、数据库连接、Knife4j、CORS | 前端和两个后端服务可启动，接口文档可访问 |
| 第 2 阶段 | 微服务基础能力 | P0 | 统一响应、错误码、全局异常、服务间调用 Client、AI 降级 | `diagnosis-service` 能调用 `ai-service` Mock 接口 |
| 第 3 阶段 | 认证与基础数据 | P0 | 患者注册登录、医生登录、科室和医生初始化数据、JWT、Axios 封装 | 患者和医生能分别登录对应入口 |
| 第 4 阶段 | 传统挂号业务 | P0 | 医生列表、医生详情、挂号创建、挂号列表、取消挂号 | 患者可完成在线挂号并查看记录 |
| 第 5 阶段 | 智能分诊微服务联调 | P0 | `ai-service` 分诊接口、业务服务转发、分诊记录、推荐医生跳转挂号 | 输入主诉后能跨服务获得推荐并进入挂号 |
| 第 6 阶段 | 医生工作台和病历 | P0 | 待诊患者、AI 病历生成、病历保存、病历查询 | 医生可生成、编辑、保存病历 |
| 第 7 阶段 | 处方与审核 | P0 | 处方开具、AI 审核、审核记录、处方查询 | 医生可审核并保存处方，患者可查看 |
| 第 8 阶段 | 综合联调和演示 | P0 | 端到端演示、Postman 记录、截图或录屏 | 固定病例能完整跑通 |
| 第 9 阶段 | WebSocket 实时通知 | P0 | 高风险处方实时告警 | 医生端无需刷新收到通知 |
| 第 10 阶段 | AI 流式响应 | P0 | SSE 或 WebSocket 病历生成流式接口、前端逐字展示 | 生成过程可见，失败可恢复 |
| 第 11 阶段 | 前端状态机优化 | P0 | Pinia 模块化状态、流程状态枚举 | AI 请求、挂号、病历、处方状态可控 |
| 第 12 阶段 | Prompt 工程 | P0 | Prompt 模板、任务类型、科室模板 | 可切换通用模板和心内科模板 |
| 第 13 阶段 | 双端分离部署 | P0 | Vue `dist`、Spring Boot Jar、Nginx 代理 | 通过部署环境演示完整流程 |
| 第 14 阶段 | 微服务拆分 | P0 | 独立 AI 微服务、Spring Cloud Feign 调用 | 主业务后端可通过 Feign 调用 AI 微服务 |

## 3. MVP 范围

MVP 必须实现：

| 模块 | 必做功能 |
|---|---|
| 基础框架 | Spring Boot 3、Vue 3、MySQL、Knife4j、CORS、统一响应 |
| 微服务架构 | `diagnosis-service`、`ai-service`、Spring Cloud Feign 调用、AI 降级 |
| 认证 | 患者注册登录、医生登录、JWT 鉴权、角色识别 |
| 基础数据 | 科室、医生、药品示例数据 |
| 挂号 | 医生列表、挂号创建、我的挂号、取消挂号 |
| 智能分诊 | 主诉输入、`diagnosis-service` 调用 `ai-service` 推荐科室和医生、记录入库 |
| 病历 | 医生录入问诊文本、调用 `ai-service` 生成病历、医生确认保存 |
| 处方 | 医生录入药品、调用 `ai-service` 审核、风险提示、确认保存 |
| 查看记录 | 患者查看本人挂号、病历、处方 |
| 联调 | Swagger/Postman 记录、前端操作截图、演示脚本 |
| 六项必做任务 | WebSocket 通知、流式响应、Pinia 状态机、Prompt、分离部署、AI 微服务 |
| 成果材料 | 汇报 PPT、数据库设计文档、接口测试记录、前端截图或演示录屏 |

MVP 暂不做：

- 真实支付、医保、检查检验系统。
- 管理员完整后台。
- 真实医院生产数据。
- 复杂排班算法。
- AI 自动决策保存病历或处方。

## 4. 详细开发任务

### 4.1 后端任务

| 任务 | 说明 | 依赖 |
|---|---|---|
| 创建微服务项目 | 创建 `diagnosis-service` 和 `ai-service`，引入 Web、Validation、Lombok、Knife4j | 无 |
| 配置数据库 | `application.yml` 配置 MySQL、JPA、时区 | MySQL |
| 统一响应 | `Result<T>`、错误码、分页响应 | 无 |
| 全局异常 | 参数异常、业务异常、权限异常、AI 异常统一处理 | 统一响应 |
| 服务间调用 | 在 `diagnosis-service` 中封装 `AiTriageClient`、`AiMedicalRecordClient`、`AiPrescriptionCheckClient` | `ai-service` |
| JWT 鉴权 | 登录签发 Token，请求拦截校验角色 | 用户表 |
| 患者接口 | 注册、登录、个人信息 | JWT |
| 医生接口 | 登录、列表、详情 | 科室和医生表 |
| 挂号接口 | 创建、列表、取消、重复挂号校验 | 患者、医生 |
| AI 分诊 | `diagnosis-service` 调用 `ai-service`，业务服务负责记录保存 | 医生、挂号、服务间调用 |
| 病历接口 | 生成、保存、列表、详情，生成能力由 `ai-service` 提供 | 挂号、服务间调用 |
| 处方接口 | 审核、创建、列表，审核能力由 `ai-service` 提供 | 病历、药品明细、服务间调用 |
| 权限校验 | 患者仅看本人数据，医生仅看接诊相关数据 | JWT |

### 4.2 AI 微服务任务

| 任务 | 说明 | 依赖 |
|---|---|---|
| AI 服务骨架 | 创建 `ai-service`，提供独立端口、Knife4j 和健康检查 | 无 |
| 分诊内部接口 | `POST /internal/ai/triage`，输入主诉，返回推荐科室、医生建议和理由 | Mock AI |
| 病历生成内部接口 | `POST /internal/ai/medical-record/generate`，输入问诊文本，返回结构化病历 | Mock AI |
| 处方审核内部接口 | `POST /internal/ai/prescription/check`，输入患者信息和药品明细，返回风险等级和建议 | Mock AI |
| Prompt 模板 | 按任务类型维护通用模板，并提供儿科、心内科等科室模板 | AI 接口 |
| Provider 适配 | 先实现 Mock Provider，再预留真实大模型 Provider | AI 接口 |
| 降级响应 | 外部 AI 超时或异常时返回可解释的降级结果 | Provider |

| 任务 | 说明 | 依赖 |
|---|---|---|
| 创建 Vue 项目 | Vue 3、TypeScript、Vite、Element Plus | 无 |
| Axios 封装 | Token 注入、错误处理、超时提示 | 后端响应约定 |

| Pinia 基础状态 | 用户信息、Token、角色 | 登录接口 |
| 登录注册页面 | 患者注册、患者/医生登录 | 认证接口 |
| 患者首页 | 展示智能分诊、挂号、记录入口 | 路由 |
| 智能分诊页 | 主诉输入、推荐结果、去挂号 | 分诊接口 |
| 在线挂号页 | 科室筛选、医生列表、时间段、提交挂号 | 医生和挂号接口 |
| 医生工作台 | 待诊患者列表、进入问诊和处方 | 挂号列表 |
| 问诊病历页 | 对话录入、AI 生成、表单回填、保存 | 病历接口 |
| 处方开具页 | 药品明细、AI 审核、风险展示、保存 | 处方接口 |
| 患者记录页 | 挂号、病历、处方列表和详情 | 查询接口 |

### 4.4 AI 集成任务

| AI 能力 | 输入 | 输出 | 降级策略 |
|---|---|---|---|
| 智能分诊 | 患者主诉、基础信息 | 推荐科室、医生、原因 | `ai-service` 返回降级结果，前端允许手动选择科室和医生 |
| 病历生成 | 医患对话、挂号信息 | 结构化病历字段 | `ai-service` 返回空草稿或失败原因，前端保留手动填写表单 |
| 处方审核 | 患者信息、药品明细 | 风险等级、相互作用、建议 | `ai-service` 返回人工复核建议，医生确认并记录未审核原因 |

AI 接口要求：

- AI 能力统一封装在 `ai-service`，不允许前端或业务 Controller 直接调用大模型。
- `diagnosis-service` 只调用 `ai-service` 的内部接口。
- 尽量要求 AI 输出 JSON。
- 后端校验 AI 输出字段，不合法则返回 `600` 或降级结果。
- 保存 AI 原始结果摘要，便于答辩和调试。

## 5. 六项必做任务开发规划

### 5.1 必做任务顺序

| 顺序 | 必做任务 | 预计投入 | 依赖 | 答辩价值 |
|---|---|---|---|---|
| 1 | WebSocket 实时通知 | 中 | 处方审核 | 展示高风险用药实时告警 |
| 2 | AI 流式响应 | 中高 | 病历生成 | 展示 AI 生成过程和交互体验 |
| 3 | 前端状态机优化 | 中 | 核心页面 | 展示复杂流程可控 |
| 4 | Prompt 工程优化 | 中 | AI Service | 展示 AI 工程化能力 |
| 5 | 双端分离部署 | 中 | 全流程稳定 | 展示工程交付 |
| 6 | 微服务拆分 | 高 | AI Service 稳定 | 展示架构进阶和服务间调用 |

### 5.2 WebSocket 实时通知

- 后端配置 WebSocket 连接路径 `/ws/notifications`。
- 医生登录后建立连接并绑定 doctorId。
- AI 处方审核返回 `HIGH` 时，后端向医生端推送高风险告警。
- 前端医生端展示通知弹层或右侧通知列表。

验收：

- 高风险处方审核后医生端无需刷新即可收到提醒。
- 非处方所属医生收不到该提醒。
- WebSocket 断开不影响处方保存主流程。

### 5.3 AI 流式响应

- 病历生成接口支持 SSE 或 WebSocket 流式输出。
- 后端输出 `start`、`delta`、`structured`、`done`、`error` 等事件。
- 前端逐字或逐段展示 AI 生成内容。
- 保留非流式病历生成接口作为兜底。

验收：

- 病历生成时前端可持续展示生成内容。
- 流式失败时显示错误并保留手动填写入口。
- 原有病历生成和保存流程不受影响。

### 5.4 前端状态机优化

- 新增 `useTriageStore`、`useConsultationStore`、`usePrescriptionStore`。
- 使用 Pinia 模块化管理患者信息、医生信息、挂号记录、处方记录等状态。
- 为分诊、病历、处方定义流程状态枚举。
- AI 请求期间禁用重复提交按钮。

验收：

- 分诊失败后可手动挂号。
- 病历生成失败后可手动填写。
- 处方审核高风险时页面明确提示并要求确认。

### 5.5 Prompt 工程优化

- 后端 Service 设计可配置 Prompt 模板。
- 支持按任务类型和科室维度选择模板。
- 至少提供通用模板、儿科模板、心内科模板和处方审核模板。
- AI 输出 JSON Schema 写入文档或配置。

验收：

- 不改业务代码即可调整 Prompt 文案。
- 能说明不同科室模板的差异。
- AI 输出异常时后端能识别并降级。

### 5.6 双端分离部署

- 前端执行 `npm run build` 生成 `dist`。
- 前端部署为 Nginx 静态资源。
- 后端执行 `mvn clean package` 生成可独立运行 Jar 包。
- Nginx 代理 `/api` 到后端服务。

验收：

- 浏览器通过 Nginx 地址访问系统。
- `/api` 请求能正确代理到后端。
- 演示病例能在部署环境跑通。

### 5.7 微服务拆分

- 将 AI 能力层抽取为独立微服务。
- 主业务后端通过 Spring Cloud Feign 调用 AI 微服务。
- AI 微服务负责智能分诊、病历生成、处方审核、Prompt 模板和 AI Provider 适配。
- AI 微服务不可用时，主业务后端返回降级提示，不影响手动流程。

验收：

- 主业务后端和 AI 微服务可以分别启动。
- Spring Cloud Feign 调用链路可通过接口测试验证。
- AI 微服务关闭时，主业务流程仍能降级运行。

## 8. 联调计划

联调顺序：

1. 先验证 `ai-service` 三个内部 AI 接口。
2. 再验证 `diagnosis-service` 调用 `ai-service` 的 Client 和降级逻辑。
3. 通过 Knife4j 或 Postman 验证业务服务对外接口。
4. 前端接入登录和 Token。
5. 前端接入医生列表和挂号。
6. 前端接入智能分诊。
7. 前端接入病历生成和保存。
8. 前端接入处方审核和保存。
9. 执行完整病例演示。

每个接口联调需记录：

- 请求路径、方法、参数。
- 成功响应截图。
- 失败响应截图。
- 前端页面操作截图。
- 异常或降级处理说明。

## 9. 演示脚本

推荐固定病例：

```text
患者：王小明，男，35 岁。
主诉：胸痛伴气短，活动后加重两天。
AI 分诊：推荐心内科。
挂号医生：张医生，心内科主任医师。
问诊对话：患者胸痛伴气短两天，活动后加重，无明确药物过敏史。
AI 病历：生成主诉、现病史、初步诊断和治疗建议。
处方：阿司匹林 100mg 每日一次 口服。
AI 审核：提示胃肠道风险，建议饭后服用并关注出血风险。
```

演示步骤：

1. 患者注册并登录。
2. 输入主诉并触发 AI 分诊。
3. 根据推荐医生完成挂号。
4. 医生登录并查看待诊患者。
5. 医生录入问诊对话并生成病历。
6. 医生确认并保存病历。
7. 医生开具处方并触发 AI 审核。
8. 医生确认保存处方。
9. 患者查看自己的挂号、病历和处方。
10. 展示 WebSocket 告警、流式生成、Pinia 状态流转、Prompt 模板、分离部署和 AI 微服务调用。

## 11. 答辩准备

答辩时建议按以下顺序讲解：

1. 为什么采用前后端分离架构。
2. Spring Boot 3 微服务如何拆分，Vue 3 负责哪些前端能力。
3. 挂号、病历、处方三个传统业务如何串联。
4. 智能分诊、病历生成、处方审核分别嵌入在哪些业务节点。
5. AI 请求如何处理 loading、超时、失败和降级。
6. 数据库如何同时支撑传统业务记录和 AI 调用记录。
7. `diagnosis-service` 与 `ai-service` 的服务边界、调用链、降级策略。
8. 六项必做任务如何落地：WebSocket、流式输出、状态机、Prompt、双端部署和微服务拆分。
