# 需求分析文档

## 1. 功能需求

| 模块 | 功能点 | 说明 |
|---|---|---|
| 认证模块 | 患者注册 | 患者填写姓名、手机号、密码、基础信息完成注册 |
| 认证模块 | 患者登录 | 使用手机号和密码登录，成功后返回 JWT |
| 认证模块 | 医生登录 | 医生使用预置账号登录医生端 |
| 患者管理 | 个人信息 | 患者查看本人基础信息 |
| 医生管理 | 医生列表 | 支持按科室筛选医生 |
| 医生管理 | 医生详情 | 查看医生职称、擅长方向、所属科室 |
| 挂号管理 | 创建挂号 | 患者选择医生和时间段完成挂号 |
| 挂号管理 | 挂号列表 | 患者查看本人挂号，医生查看自己接诊列表 |
| 挂号管理 | 取消挂号 | 患者取消未就诊挂号 |
| 智能分诊 | AI 分诊 | 输入主诉后推荐科室和医生 |
| 智能分诊 | 分诊记录 | 保存主诉、推荐结果、患者、时间 |
| 病历管理 | AI 生成病历 | 根据医患对话生成结构化病历草稿 |
| 病历管理 | 保存病历 | 医生确认或修改后保存病历 |
| 病历管理 | 病历查询 | 患者查看本人病历，医生查看相关病历 |
| 处方管理 | 开具处方 | 医生录入药品、剂量、频次、用法 |
| 处方管理 | AI 审核 | 审核药品相互作用、禁忌和风险 |
| 处方管理 | 处方查询 | 患者和医生查询相关处方 |
| 管理端 | 管理员登录/权限 | 管理员通过独立入口登录，访问 `/api/admin/*` 需要 ADMIN 角色 |
| 管理端 | 科室管理 | 维护科室名称、描述、启停状态 |
| 管理端 | 医生管理 | 维护医生账号、科室归属、职称和擅长方向 |
| 管理端 | 药品管理 | 维护药品名称、禁忌、相互作用基础规则 |
| 管理端 | Prompt 模板管理 | 维护分诊、病历生成、处方审核模板 |
| 管理端 | 系统字典管理 | 维护性别、状态、任务类型、科室编码、风险等级等字典 |
| 管理端 | AI 医生排班 | 选择科室、日期范围、医生和号源规则，生成 AI 排班建议并人工确认发布 |
| 管理端 | AI 分诊台 | 查看分诊记录、AI 推荐科室/医生、推荐理由和处理状态，支持人工改派 |

## 2. 非功能需求

| 类型 | 要求 |
|---|---|
| 可运行性 | 前端通过 `npm run dev` 启动，各后端微服务通过 `mvn spring-boot:run` 分别启动 |
| 可演示性 | 浏览器中可完整演示患者注册到医生开方流程 |
| 代码规范 | 各后端微服务遵循 Controller-Service-Repository 分层，前端使用 Vue 3 组合式 API |
| 架构一致性 | 后端从项目初始化起按业务域划分为独立微服务，各服务拥有清晰职责和数据边界 |
| WebSocket 实时通知 | AI 审核发现高风险用药时，医生端必须收到实时告警 |
| AI 流式响应 | 病历生成必须支持 SSE 或 WebSocket 流式输出 |
| 状态机优化 | 前端必须使用 Pinia 模块化管理患者、医生、挂号、处方等关键状态 |
| Prompt 工程 | 后端必须支持可配置 Prompt 模板，并支持按科室优化 |
| 分离部署 | 前端必须可部署为 Nginx 静态资源，各后端微服务必须可分别打包为独立 Jar |
| 纯微服务要求 | 必须具备 Gateway、注册发现、业务域服务划分、OpenFeign 调用、熔断降级和独立数据边界 |
| 体验友好 | AI 请求展示 Loading，失败时给出明确提示 |
| 稳定性 | AI 服务异常时允许回退到手动选择、手动填写、手动保存 |
| 安全性 | JWT 鉴权，角色权限控制，数据归属校验 |
| 可维护性 | 接口使用 Swagger/Knife4j 自动生成文档 |

## 3. 用户角色

| 角色 | 权限 |
|---|---|
| 游客 | 注册、登录、查看公开医生列表 |
| 患者 | 智能分诊、在线挂号、取消挂号、查看本人病历和处方 |
| 医生 | 查看本人挂号患者、生成和保存病历、开具和审核处方 |
| 管理员 | 维护医生、科室、药品、Prompt 模板、AI 配置、系统字典，处理 AI 医生排班和 AI 分诊台 |

## 4. 用户故事

1. 作为患者，我希望输入症状后获得推荐科室和医生，减少盲目挂号。
2. 作为患者，我希望在线选择医生和时间段完成挂号。
3. 作为患者，我希望随时查看自己的挂号记录、电子病历和处方记录。
4. 作为医生，我希望查看今日待接诊患者，快速进入问诊。
5. 作为医生，我希望输入医患对话后自动生成结构化病历草稿。
6. 作为医生，我希望在保存处方前获得 AI 用药风险提示。
7. 作为学生开发者，我希望通过该项目理解前后端分离、Spring Cloud 纯微服务、数据库建模和 AI 融合开发。
8. 作为管理员，我希望维护科室、医生、药品、系统字典和 Prompt 模板，保证患者端与医生端流程有稳定基础数据。
9. 作为管理员，我希望使用 AI 生成医生排班建议，并在人工确认后发布号源。
10. 作为管理员，我希望在 AI 分诊台查看分诊记录并人工改派异常推荐，保证挂号流程可控。

## 5. 业务流程

```mermaid
flowchart LR
  A[患者注册登录] --> B[输入症状主诉]
  B --> C[AI 智能分诊]
  C --> D[推荐科室和医生]
  C --> X[管理端 AI 分诊台复核/改派]
  X --> D
  Y[管理端 AI 排班发布号源] --> E
  D --> E[在线挂号]
  E --> F[医生查看待诊患者]
  F --> G[录入问诊内容]
  G --> H[AI 生成病历草稿]
  H --> I[医生确认保存病历]
  I --> J[医生开具处方]
  J --> K[AI 处方审核]
  K --> L[医生确认保存处方]
  L --> M[患者查看诊疗记录]
```

## 6. 边界情况

| 场景 | 处理规则 |
|---|---|
| 主诉为空或过短 | 禁止提交，提示补充症状描述 |
| AI 分诊失败 | 提示“AI 服务暂不可用”，允许手动选择科室 |
| 医生无可挂号时间段 | 显示空状态，不允许提交 |
| 重复挂号 | 同一患者同一医生同一时间段不可重复挂号 |
| 取消已完成挂号 | 已完成或已就诊挂号不可取消 |
| 处方药品为空 | 不可保存处方 |
| AI 审核高风险 | 提示风险，允许医生修改或确认后保存，并记录确认行为 |
| 病历生成失败 | 保留手动填写入口 |
| AI 排班失败 | 管理员可手动维护排班和号源 |
| AI 分诊推荐不合理 | 管理员可在 AI 分诊台人工改派科室或医生 |
| JWT 过期 | 清理本地登录态，跳转登录页 |

## 7. 约束条件

- 使用 Spring Boot 3 + Spring Cloud + Vue 3 + TypeScript。
- 前端只通过 `gateway-service` 暴露的 RESTful API 通信。
- 使用 JWT 鉴权。
- 使用 MySQL 或 PostgreSQL，MVP 推荐 MySQL 8。
- 前端只访问 `gateway-service`，不直接调用业务服务或 AI 服务。
- 后端按认证、患者、医生、挂号、分诊、病历、处方、AI、通知等业务域划分为独立微服务。
- 管理端是第三个必做入口，覆盖基础数据维护、AI 医生排班和 AI 分诊台。
- 服务之间通过 OpenFeign 调用，并通过注册发现、超时、熔断和降级保证链路稳定。
- AI 输出必须经医生确认后才能保存为正式病历或处方审核依据。
- 项目使用模拟数据，不接入真实医疗机构生产数据。
