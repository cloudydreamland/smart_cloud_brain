# 需求分析文档

## 1. 功能需求

| 模块 | 功能点 | 说明 |
|---|---|---|
| 认证 | 患者登录 | 使用手机号和密码登录，返回 JWT |
| 认证 | 医生登录 | 使用医生账号或手机号登录医生端 |
| 认证 | 管理员登录 | 使用管理员账号登录管理端 |
| 患者 | 个人信息 | 查看患者基础资料 |
| 分诊 | AI 分诊 | 输入主诉后推荐科室、医生和就诊建议 |
| 分诊 | 分诊记录 | 保存主诉、推荐结果、患者和时间 |
| 挂号 | 号源列表 | 从数据库读取可预约号源 |
| 挂号 | 创建挂号 | 患者选择号源后生成挂号记录 |
| 挂号 | 取消挂号 | 患者取消自己的未就诊挂号 |
| 医生 | 待接诊队列 | 医生查看本人名下挂号 |
| 病历 | AI 生成病历 | 基于挂号和主诉生成病历草稿 |
| 病历 | 保存病历 | 医生确认或修改后保存正式病历 |
| 处方 | AI 处方审核 | 审核药品禁忌、相互作用和风险 |
| 处方 | 保存处方 | 医生确认后保存处方和明细 |
| 通知 | 风险通知 | RabbitMQ 事件触发 WebSocket 通知 |
| 管理 | 基础数据 | 维护科室、医生、药品、Prompt、知识库、字典 |
| 管理 | AI 排班 | 生成排班建议并人工确认发布 |
| 管理 | 分诊工作台 | 查看分诊记录、改派医生、关闭记录 |

## 2. 非功能需求

- 可演示：核心流程可以在本机或 Docker 环境完整跑通。
- 可验证：业务数据可在数据库中查询确认。
- 可维护：服务边界清晰，公共契约兼容优先。
- 可降级：AI 和消息队列异常时，主业务有可说明的降级策略。
- 安全性：JWT 鉴权、角色控制、患者和医生数据归属校验。
- 可部署：支持 Docker Compose 一键启动。

## 3. 数据需求

- 基础数据：患者、医生、管理员、科室、药品、Prompt、知识库、字典。
- 号源数据：医生排班和可预约号源。
- 交易数据：挂号、分诊、病历、处方、通知。
- AI 数据：Prompt 模板、AI 调用日志、结构化结果摘要。

所有核心数据均来自 KingbaseES/PostgreSQL 兼容数据库，前端不维护业务假数据。

## 4. AI 需求

AI 当前使用 Mock Provider，但必须满足：

- 由后端 `ai-service` 提供能力。
- 业务服务通过后端链路调用 `ai-service`。
- 前端不伪造分诊、病历、处方审核或排班结果。
- AI 输出只作为辅助建议，最终由医生或管理员确认。

## 5. 边界说明

不做真实支付、医保、住院、药房库存、真实发药、真实医院生产数据接入和真实医疗诊断。
