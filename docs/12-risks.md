# 风险与问题清单

## 1. 需求风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 医疗业务范围过大 | 开发周期失控 | MVP 只覆盖分诊、挂号、病历、处方 |
| 管理端范围扩大 | 影响开发周期 | 管理端必做范围限定为科室、医生、药品、Prompt 模板和系统字典维护 |
| 真实医疗合规要求复杂 | 不适合课程项目完整落地 | 明确系统仅用于教学演示 |

## 2. 技术风险

| 风险 | 影响 | 建议 |
|---|---|---|
| AI 返回格式不稳定 | 前端无法正确展示 | 后端要求 JSON 输出并做字段校验 |
| AI 服务超时 | 页面长时间等待 | 设置超时、Loading 和失败降级 |
| 微服务调用失败 | AI 能力不可用或业务流程中断 | `diagnosis-service` 为每个 AI 调用设置超时、降级和日志 |
| 服务边界不清 | 业务逻辑和 AI 逻辑重复实现 | 业务数据归 `diagnosis-service`，模型调用和 Prompt 归 `ai-service` |
| 前后端联调问题多 | 开发效率下降 | 先通过 Knife4j/Postman 验证接口 |
| JWT 权限实现不完整 | 数据越权访问 | 加角色校验和数据归属校验 |
| 数据库关系复杂 | 查询和保存容易出错 | 先实现核心表，复杂关系后续迭代 |

## 3. 数据风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 患者隐私敏感 | 不适合公开展示 | 使用模拟数据 |
| 演示数据不完整 | 演示流程中断 | 准备固定初始化 SQL |
| AI 记录过大 | 数据库膨胀 | 保存摘要和必要 JSON，定期归档 |

## 4. 安全风险

| 风险 | 影响 | 建议 |
|---|---|---|
| Token 泄露 | 账号被冒用 | 设置过期时间，生产使用 HTTPS |
| 明文密码 | 严重安全问题 | 使用 BCrypt 等方式哈希存储 |
| 越权访问病历处方 | 隐私泄露 | 后端每个查询都校验数据归属 |
| AI Key 泄露 | 产生费用和安全风险 | 使用环境变量，不提交到代码仓库 |

## 5. 时间风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 模块过多 | 无法按时完成 | 先跑通诊疗主线，再逐项完成六项必做任务 |
| AI 集成耗时 | 阻塞主流程 | 先使用 Mock AI 服务 |
| 前端页面过多 | UI 开发时间不足 | 复用布局和表格表单组件 |
| 部署环境差异 | 答辩现场无法运行 | 提前准备本地演示版本和录屏 |

## 6. 对应解决建议

1. 第一优先级完成传统业务闭环，保证患者登录、医生列表、挂号流程可用。
2. AI 能力先 Mock 后接真实服务，避免 AI 接入阻塞主线。
3. 每完成一个接口即通过 Knife4j 或 Postman 测试。
4. 前端优先实现可演示流程，不在早期投入复杂视觉优化。
5. 所有 AI 结果均标注为辅助建议，并要求医生确认。
6. 提前准备演示数据、演示脚本和备用录屏。
