# 风险与问题清单

## 1. 需求风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 医疗业务范围过大 | 开发周期失控 | MVP 只覆盖分诊、挂号、病历、处方、通知和管理端基础能力 |
| 管理端范围膨胀 | 影响主流程交付 | 管理端 P0 限定为基础数据、AI 排班、分诊工作台和字典维护 |
| 真实医疗合规复杂 | 不适合课程项目完整落地 | 明确系统仅用于教学演示，不作为真实诊疗依据 |
| AI 能力被误解为真实诊断 | 答辩口径不清 | 说明 AI 当前为 Mock Provider，输出仅作辅助建议 |

## 2. 技术风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 前端绕过后端伪造结果 | 演示真实性不足 | 三端统一走 Gateway，禁止前端写死业务结果 |
| AI 服务不可用 | 分诊、病历、处方审核受影响 | 保留 Mock Provider 和降级结果 |
| 微服务调用失败 | 主链路中断 | 配置超时、错误日志、幂等和可恢复状态 |
| RabbitMQ 不可用 | 风险通知延迟 | 主事务先落库，Outbox 等待重试 |
| WebSocket 断线 | 医生端实时性下降 | 前端重连，同时保留通知历史查询 |
| 权限校验不完整 | 数据越权访问 | 网关校验角色，服务内校验数据归属 |

## 3. 数据风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 演示数据不完整 | 现场流程中断 | 使用 `sql/kingbase_schema.sql` 初始化基础数据和号源 |
| 使用前端假数据 | 无法证明真实流程 | 所有核心数据从 KingbaseES 查询或由流程写入 |
| 数据库脚本乱码 | 初始化后页面显示异常 | SQL 文件统一 UTF-8，演示前执行抽样查询 |
| AI 日志过大 | 数据库膨胀 | 保存摘要和必要 JSON，后续可定期归档 |

## 4. 部署风险

| 风险 | 影响 | 建议 |
|---|---|---|
| Maven 依赖缺失 | 离线编译失败 | 固定本地仓库 `D:\DEVELOP\maven`，新增依赖先联网下载 |
| Docker 镜像下载慢 | 现场启动失败 | 提前拉取镜像并准备录屏 |
| 端口冲突 | 页面或服务无法访问 | 检查 `18080`、`5173`、`5174`、`5175`、`8848`、`15672` |
| 外部数据库配置错误 | 服务无法连库 | 优先使用 Compose 内置兼容数据库完成演示 |

## 5. 答辩口径风险

答辩时需要主动说明：

1. 真实的是业务链路、微服务调用、数据库读写、RabbitMQ/WebSocket 通知。
2. Mock 的是大模型 Provider 输出，前端没有伪造 AI 结果。
3. AI 结果必须经过医生或管理员确认，不能作为真实医疗诊断。
4. 系统重点展示工程完整性，而不是医疗诊断准确率。
